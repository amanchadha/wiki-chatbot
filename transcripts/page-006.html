# Design Rationale

A Wikipedia-grounded question-answering system built as a **loop-engineering
exercise**: rather than hand-tuning a system prompt up front, the eval suite
was built as the feedback signal and the prompt was iterated against it.
Every prompt change in this project was motivated by a specific, transcribed
failure from an eval run, and every change's effect was predicted before the
run and measured after it.

**Final numbers** (40-case suite, final config `v1.3`): pass rate 1.0 and
0.975 across two consecutive runs of the identical config; search recall
0.955; search precision 1.00 on every one of the project's runs; right-article
rate 1.0; zero faithfulness violations; zero tool errors.

> **Transparency note, up front:** the 40/40 run followed two **eval-side
> grading calibrations** made after the first final-config run (39/40). The
> system prompt, tool, and model were byte-identical in both runs. The two
> calibrations were: (1) case r09's notes now accept either 1906 or 1907 for
> Longyearbyen's founding — the Wikipedia article itself is internally
> inconsistent (prose says 1906, infobox says 1907), and the agent had
> faithfully reported an evidence-supported value; (2) the
> `wrong_abstention_ids` metric no longer flags the dead-end multi-hop case
> whose *designed correct behavior* is abstention. Both are grading fixes,
> not behavior fixes — the honest headline is "39–40 of 40, with the one
> disputed grade caused by Wikipedia disagreeing with itself."

## System design

- **Agent**: `claude-opus-4-8`, adaptive thinking, `effort: low`, a manual
  tool-use loop (~40 lines, no SDK runner — the mechanics stay visible), one
  tool: `search_wikipedia(query)`. Opus 4.8 was chosen because the
  search/no-search decision is the core prompt-engineered behavior and its
  prompt-steered tool triggering is the most faithful; it also rejects
  sampling parameters, which helps eval reproducibility.
- **Tool**: live MediaWiki API — search titles + snippets, plain-text extract
  of the best hit, and the article's infobox fields (see "retrieval
  hardening" below), with process-wide throttling and 429/503 backoff.
- **Judge**: `claude-sonnet-5` — deliberately a different model from the
  agent to avoid self-grading bias — with schema-enforced JSON verdicts and a
  version-tagged rubric prompt.

## Eval design

**Three graded dimensions**, chosen so every failure decomposes along the
pipeline: **search decision** (code-graded: did it search when it should?),
**retrieval quality** (judge-graded: did the tool return the needed
evidence, from the right article?), **answer quality** (judge-graded:
correct? faithful to the evidence? honest when it should abstain?). The
triage chain — wrong+no-search = decision failure, wrong+insufficient
evidence = retrieval failure, wrong+sufficient evidence = generation
failure — told each iteration exactly which layer to fix.

**Case design** (40 cases, 6 categories): stale-memory traps where a
confident wrong answer is the *anticipated* failure (post-training
officeholders, grown counts); no-search cases to catch over-searching;
three-valued `expect_search` (true/false/optional, optionals excluded from
precision/recall); forced-disambiguation pairs ("Mercury Records" vs the
planet); unanswerables where honesty is the pass condition; and multi-hop
chains with obscure intermediates that memory can't shortcut, including one
deliberate dead-end where the right answer is "that isn't recorded."

**Judge design**: per-notch examples for every rubric value; an
authoritative per-case `notes` field that overrides the judge's own
knowledge (so trap cases grade correctly); instructions to flag — but not
act on — disagreement with the expected answer. A 10-case blind human
spot-check, selected before any verdicts existed, matched the judge 10/10.
Judge prompt changes are version-tagged in every summary row, and a
`--rejudge` mode replays a new judge over frozen transcripts so grading
improvements never require re-running (or re-randomizing) the agent.

**Lexical-metrics experiment**: BLEU/ROUGE/BERTScore/MoverScore were run as
a sidecar and tested for agreement with judge verdicts. BLEU and ROUGE were
dropped on the evidence — on short factual answers, a fluent wrong answer
("Saturn has 146 confirmed moons") overlaps the reference as well as a right
one; the baseline's sole incorrect answer scored *above* the correct-group
ROUGE-1 mean, and correct answers spanned 0.0–0.76 so no threshold separates
the groups. They also punish honest abstention, the exact behavior the suite
rewards. MoverScore never built; BERTScore is deferred (network), and the
LLM judge is the answer-quality signal of record.

Three themes ran through every iteration:

1. **Confidence is not correctness.** v0's prompt keyed searching on the
   model's own certainty; the baseline showed that certainty is miscalibrated
   on exactly the volatile facts that matter. v1 re-keyed the decision on the
   *shape of the fact* (current / count / named-specific), checkable from the
   question alone — recall tripled while the no-search list held precision at
   1.0. The same instinct resurfaced twice more (answering beyond retrieved
   evidence; backfilling later hops of a chain) and was closed the same way:
   explicit evidence-grounding rules, never "search more."
2. **Retrieval hardening was about page furniture, not search.** Plain-text
   extracts silently strip wikitables *and* infoboxes — award winners and
   settlement populations live only there. Search snippets (drawn from the
   indexed page text) and extracted infobox fields closed both gaps; the
   right-article rate was never the problem.
3. **Instrument the infrastructure separately.** The 429 outage run polluted
   the judged retrieval metrics until `tool_error_rate` (code-counted)
   separated "the API refused us" from "we retrieved the wrong page." Metrics
   need the same failure-attribution discipline as the system under test.

## Where it still fails, knowingly

- **r08 noise band** (recall 0.955, not 1.0): stable decades-old historical
  years ("Vanuatu, 1980") sometimes get answered from memory — correctly, in
  every observed run. Tightening the trigger wording risks the 1.0 precision
  streak; deliberately left.
- **Unsupported side details** (~5% of searched answers, measured by
  judge-v4): tiny memory embellishments on otherwise-grounded answers — a
  "(France/PSG)" affiliation, a "(12 August 1970)" day-and-month where
  evidence gave the year. True facts in both observed instances, but
  unverified specifics of exactly the class that eventually goes wrong. Now
  tracked in every run; a one-line prompt remedy is drafted if the rate
  climbs on a larger case set.
- **The dead-end case reads as "insufficient evidence"** in the retrieval
  metric — by design; its abstention is the pass condition.
- **Small, live-data eval**: n=40, graded by an LLM (blind-spot-checked
  once), against live Wikipedia whose facts drift. Case notes push grading
  toward "against retrieved evidence" to absorb drift, but a production
  version wants a pinned corpus and a larger, held-out case set.

## What the evals taught that hand-tuning wouldn't have

The baseline's scariest number was invisible in its pass rate: 0.94 looked
fine while search recall was 0.33 — nine latent failures passing on memory
that happened to be right. Every meaningful discovery of the project (chain
collapse, table/infobox blindness, the outage behavior fork, the side-claims
gap) came from reading *transcripts of passes*, not just failures. The
metrics that made iteration honest were the ones that refused to let a lucky
pass hide a broken mechanism.
